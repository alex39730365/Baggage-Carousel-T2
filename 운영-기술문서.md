# 수하물 캐러셀 현황판 운영/기술 문서

## 1) 프로젝트 개요

`수하물 캐러셀 현황판`은 인천국제공항공사 공공데이터 API를 기반으로,  
도착 항공편의 수하물 캐러셀/도착 시각/처리 시간을 1분 단위로 확인하는 내부 참고용 웹 도구입니다.

주요 목적:

- 현장/사무 직원이 빠르게 현재 수하물 현황을 파악
- 날짜/터미널/편명 기준 조회
- 수하물 처리시간(첫 벨트~마지막 벨트) 확인
- 다수 사용자가 동시에 접속해도 API 호출량을 억제

## 2) 기술 스택

- Frontend: `React 18`, `TypeScript`, `Vite`, `Tailwind CSS`
- Backend(API Route): Vercel Serverless Function (`api/baggage-arrivals.ts`)
- 배포: Vercel
- 데이터 소스: 인천국제공항공사 공공데이터 API  
  (`getFltArrivalsBaggageClaimDesk`)

## 3) 구동 방식 (전체 아키텍처)

흐름:

1. 브라우저가 `/api/baggage-arrivals` 호출
2. Vercel 함수가 공공데이터 API를 호출해 스냅샷 생성
3. 60초 캐시 저장 후 브라우저에 반환
4. 프론트는 반환 데이터를 정규화/병합/중복제거해서 화면 렌더링

핵심 포인트:

- 사용자가 공공데이터 API를 직접 때리지 않음
- 서버 캐시로 다수 사용자 요청을 흡수
- 클라이언트는 1분마다 재조회

## 4) API/캐시 전략

### 4.1 서버 API (`api/baggage-arrivals.ts`)

- 엔드포인트: `GET /api/baggage-arrivals`
- 내부에서 공공데이터 API 수집:
  - `searchDay = [0, 1, 2]`
  - `numOfRows = 1000`
  - 페이지는 `totalCount` 기반으로 계산 (최대 15페이지)
- 수집 결과를 하나의 스냅샷 JSON으로 합쳐 반환

### 4.2 캐시 정책

- 함수 내부 메모리 캐시 TTL: 60초
- 동시 요청 합류(in-flight dedupe):
  - 같은 시점 다중 요청은 1회 upstream 호출에 합류
- Vercel 캐시 헤더:
  - `Cache-Control: public, s-maxage=60, stale-while-revalidate=30`
- 디버그 헤더:
  - `X-Baggage-Cache: HIT | MISS | JOIN`

### 4.3 공공데이터포털 일일 트래픽(참고)

- 활용신청 기준 **일일 50,000건** 트래픽(호출 한도)
- 위 한도는 포털에서의 upstream 호출 기준이며, 본 서비스의 60초 서버 캐시·동시 요청 합류로 브라우저·Vercel 요청 수와는 별개로 실제 공공 API 호출은 상대적으로 적게 유지됨

## 5) 프론트 데이터 처리 방식

핵심 파일: `src/lib/baggageApi.ts`, `src/hooks/useBaggageData.ts`

### 5.1 1분 갱신

- `useBaggageData`에서 첫 로드 즉시 호출
- 이후 `60초` 주기로 반복 fetch

### 5.2 데이터 정규화

- 공공데이터 원본 필드명 변형 대응 (`flightId`, `flightNo`, `estimatedDatetime`, `scheduleDatetime` 등)
- 항공편 문자열 정리 (`BX165 / NRT / NRT` 같은 중복 토큰 제거)
- 캐러셀 번호 필드 여러 형태를 파싱해 숫자 배열로 통합

### 5.3 날짜/시간 보정 (중요)

- 날짜 버킷은 `scheduleDatetime` 우선 신뢰
- `estimatedDatetime` 날짜가 어긋나는 경우 날짜 누락 방지
- 서울 시간대(`Asia/Seoul`) 기준 파싱/정렬

### 5.4 중복/충돌 처리

- 동일 날짜+캐러셀+편명 기준 dedupe
- 동일 편이 여러 줄일 때 점수 기반 우선순위로 1개 선택
- 처리시간(bagFirstTime/bagLastTime) 정보는 더 풍부한 raw를 병합

## 6) 화면 기능

핵심 파일: `src/components/BaggageCarouselBoard.tsx`, `CarouselDataGrid.tsx`

### 6.1 화면 모드

- `모바일`(카드형 목록)
- `캐러셀 현황`(스케줄 격자)
- `수하물 처리 시간`(처리시간 격자)

시작 모드는 기본 `캐러셀 현황(table)`이며, 같은 탭 새로고침 시 모드를 유지합니다.

### 6.2 필터/탭

- 기준 날짜 선택
- 터미널 탭: `전체 / 터미널1 / 터미널2`
- 코드셰어 필터 기본 ON:
  - 2자+4자리 코드셰어 숨김
  - 단, `KE 2xxx/8xxx`는 예외 노출

### 6.3 검색/강조

- 편명 검색 (최대 20개 결과)
- 검색 결과 클릭 시 해당 위치 스크롤+플래시 강조
- 별표(★) 강조 저장/복원(localStorage)

### 6.4 처리시간 표시

- `bagFirstTime` ~ `bagLastTime` 기반 처리 소요(분) 계산
- 목록에서 hover 팝오버 또는 모바일 시트로 확인 가능

### 6.5 기타 UX

- 모바일 핀치 확대(격자)
- 캐러셀 10/11 사이 가이드 라인 토글
- LIVE 타임스탬프 표시(마지막 성공 갱신 시각)
- 하단 안내문(비공식/참고용 고지)

## 8) 환경변수

`.env` 예시:

- `VITE_DATA_GO_KR_SERVICE_KEY` (로컬 프록시/개발 편의)
- `DATA_GO_KR_SERVICE_KEY` (Vercel 서버 함수용)

현재 코드는 키 누락 시 fallback 키를 사용하도록 되어 있으며,  
운영 안정성을 위해 실제 키를 환경변수로 유지하는 것을 권장합니다.

## 9) 운영 시 체크 포인트 (Vercel)

- Route: `/api/baggage-arrivals`
- 확인 항목:
  - `429`, `503`, `500` 발생 추이
  - 캐시 패턴 (`HIT/STALE` 비중)
  - 응답 시간(p75/p95)
- 정상 패턴:
  - `200/304` 다수
  - `HIT/STALE` 다수
  - 타임아웃 낮음

## 10) 한계와 설계 의도

- 본 서비스는 내부 참고용이며, 원천 데이터 지연/오류 가능
- 완전한 정답 시스템이 아니라 "빠른 상황 파악" 도구를 목표로 설계
- 외부 API 한도 이슈를 피하기 위해 서버 스냅샷+캐시 중심 구조 채택
